Aim:
To flatten a multilevel doubly linked list where nodes can have a child pointer pointing to a separate doubly linked list, into a single-level doubly linked list using depth-first search order.

Definition:
Flattening a Multilevel Doubly Linked List is a structural traversal algorithm. It takes a tree-like hierarchy of doubly linked lists and flattens it in-place into a single, contiguous linear sequence.

Theory:
A multilevel doubly linked list is a data structure where each node has `prev`, `next`, and an optional `child` pointer. The child pointer points to a separate doubly linked list, which can also contain nested child lists.
Flattening this structure in depth-first search (DFS) order means that if a node has a child list, all nodes in the child list must appear before the node's next siblings.
This is similar to pre-order traversal of a multi-way tree:
`Parent -> Child -> Child's Child -> Parent's Next Sibling`
To flatten in-place, we must update the `prev` and `next` pointers to weave the child chain inline, while clearing all child pointers to NULL.

Working Principle:
1. Initialize a `curr` pointer at the head of the list.
2. Traverse the list node-by-node:
   - If the current node `curr` has a child:
     - Find the tail of the child list by traversing to the end of the child list.
     - Connect the tail of the child list to the original `curr.next` (if it exists).
     - Connect `curr.next` to the child head, and set `child.head.prev = curr`.
     - Connect the original `curr.next`'s `prev` to the child tail (if `curr.next` is not NULL).
     - Clear the child pointer: `curr.child = NULL`.
   - Move `curr` to `curr.next`.
3. Return the head of the list.

Advantages:
- In-place rewriting: Operates directly on the existing nodes without creating new list structures.
- Restores linear access: Simplifies data processing by converting a 2D hierarchical list into a 1D sequential list.

Disadvantages:
- Reference loss: Pointer updates are complex, and incorrect ordering can break the list structure, leading to lost nodes.
- High traversal overhead: Iteratively traversing to find child list tails can lead to quadratic time complexity if implemented naively (using recursion or a stack avoids this by returning the tail directly).

Applications:
- Serialization: Preparing hierarchical DOM structures or database configurations for network transmission.
- Memory compaction: Layout optimization for caches by keeping related hierarchical elements contiguous.

Real Life Applications:
- Flat layout representation of comments threads (nested replies flattened into a linear timeline).
- Serializing structured XML/JSON nodes into a flat text buffer.

Time Complexity:
O(N) where N is the total number of nodes in the multilevel list structure.

Space Complexity:
- O(1) auxiliary space for the iterative, tail-tracking method.
- O(D) stack space for recursive implementations, where D is the maximum depth of nesting.

Stable/Not Stable:
Stable. The relative order of nodes within their respective levels is preserved.

In-place/Not In-place:
In-place. Modifications are made directly on the nodes of the original list.

Prerequisites:
- Doubly linked list traversal and insertion.
- Basic understanding of stacks or DFS recursion.

Common Mistakes:
- Forgetting to clear the `child` pointer of the parent node to NULL after flattening.
- Not updating the `prev` pointer of the parent's original next sibling when linking the child list tail.
- Accessing NULL pointers when the child list tail has to be linked to a non-existent next node.

When to Use:
- When a linear, flat traversal of a hierarchical doubly linked list structure is required.
- When transforming custom tree structures into database records.

When Not to Use:
- When the hierarchical relationships must be preserved during runtime (flattening is a destructive operation that removes child references).
- When list nodes do not contain child references.
